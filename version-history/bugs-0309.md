## Bug 12：波形动画视觉效果差

**对应步骤**：⑦ 录音波形 UI

**现象**
录音时的波形动画不好看，视觉效果简陋。

**原因分析**

当前 `WaveformView`（`VoiceBarView.swift:174-188`）的实现：

```swift
HStack(alignment: .center, spacing: 2.5) {
    ForEach(levels.indices, id: \.self) { i in
        let h = CGFloat(max(0.08, levels[i]))
        Capsule()
            .fill(Color.red.opacity(0.75 + Double(h) * 0.25))
            .frame(width: 3, height: 4 + h * 28)  // 最高32pt
    }
}
```

- 30 个红色胶囊柱，最高 32pt，宽 3pt，间距 2.5pt
- 颜色单一（纯红），没有渐变或层次
- 动画只有 `.easeOut(duration: 0.08)`，过渡生硬
- 整体视觉效果像素级工具原型，不够精致

**优先级**：低

## Bug 13：识别失败时录音被删除，无法重试

**对应步骤**：⑯ 文件清理 / ⑮ 空结果判断

**现象**
用户明确说了一段话，但 App 显示"未检测到语音"，且无法重试——录音已丢失。

**根本原因**

`TranscriptionEngine.swift:157` 的 `defer` 语句：

```swift
private func transcribeFile(at url: URL) async {
    defer { try? FileManager.default.removeItem(at: url) }  // ← 无条件删除
    ...
}
```

**不管转写成功还是失败，音频文件都被立刻删除。** 这是整个流程中最大的设计缺陷。

触发"未检测到语音"的可能原因：
1. `noSpeechThreshold` 默认 0.6（60%），Whisper 认为音频的"无语音概率"超过此阈值就返回空结果
2. 录音时间过短（用户快速按松 ⌥⌘R）
3. 环境噪音导致信噪比过低
4. 模型判断失误（尤其是轻声说话或远距离录音时）

**缺失的功能：**
- 没有本地录音存储机制
- 没有可配置的保留策略（永久/30天/3天）
- 没有识别失败时的自动重试
- 没有手动重新识别的入口

**优先级**：高

## Bug 14：词汇管理窗口布局问题

**对应步骤**：词汇管理 UI

**现象**
词汇管理窗口太小，文本全部挤在一行，没有自动换行，内容显示不全。

**原因分析**

`VocabularyView.swift` 中 `TermRow`（第 56-82 行）的布局结构：

```swift
HStack {                               // ← 强制所有内容在同一行横排
    Text(term.correct).font(.body.bold())
    if !term.aliases.isEmpty {
        Text("→")
        Text(term.aliases.joined(separator: " / "))  // ← 所有别名拼成一个长字符串
            .font(.callout)
            .foregroundStyle(.secondary)
    }
}
```

具体问题：
1. **`HStack` 强制所有内容在一行** —— 当别名较多或较长时直接溢出屏幕
2. **没有 `.lineLimit(nil)` 或 `.fixedSize(horizontal: false, vertical: true)`** —— 长文本被截断而非换行
3. **窗口初始尺寸 440×480**（`AppDelegate.swift:96`）太小
4. **`AddTermView` 固定 380×380**（`VocabularyView.swift:141`）—— 添加面板也很局促
5. **箭头方向 `→` 容易误导** —— 显示为 `OpenAI → open ai`，看起来像"正确词转成错误词"，实际替换方向是反过来的

**优先级**：中

## Bug 15：自定义词汇对识别无帮助（三层 Bug）

**对应步骤**：⑪ Prompt 构建 / ⑭ 词汇替换

**现象**
用户在词汇管理中添加了内容（如正确词"OpenAI"，别名"open ai"），但后续的识别结果没有任何改善。

### Bug 15-A：词汇完全没有参与 Whisper 解码（根本原因）

整个流程中，词汇只在步骤 ⑭（事后字符串替换）被使用。但在步骤 ⑪（构建 DecodingOptions）和步骤 ⑫（Whisper 推理）中，**模型完全不知道用户的自定义词汇**。

当前的 `promptTokens` 只包含固定文本：

```swift
let promptTokens = kit.tokenizer?.encode(text: "以下是普通话的句子。")
```

**自定义词汇完全没有注入到 prompt 中。** 这意味着词汇管理对 Whisper 的识别过程零影响——它不可能帮助 Whisper "正确识别"任何词汇，只是在结果出来后做文本查找替换。

UI 描述写的是"帮助 Whisper 正确识别"，但实际实现只是事后替换，这是功能设计与实现之间的根本脱节。

**正确的做法**：将词汇注入到 `promptTokens` 中，让 Whisper 在解码时就知道这些词汇：

```swift
// 示例：动态构建 prompt
var prompt = "以下是普通话的句子。"
let vocabTerms = vocabularyStore?.terms.map { $0.correct } ?? []
if !vocabTerms.isEmpty {
    prompt += "涉及的专有名词：" + vocabTerms.joined(separator: "、") + "。"
}
let promptTokens = kit.tokenizer?.encode(text: prompt)
```

这样 Whisper 解码时上下文中包含这些词汇，会大幅提升正确输出它们的概率。

### Bug 15-B：字符串替换是大小写敏感的

`TranscriptionEngine.swift:181`：

```swift
text = text.replacingOccurrences(of: wrong, with: correct)
```

`String.replacingOccurrences(of:with:)` 默认**区分大小写**。如果用户添加别名 `open ai`，但 Whisper 输出的是 `Open AI`、`OPEN AI` 或 `Open ai`，都不会匹配。

Whisper 的大小写输出是不可预测的，同一个词多次识别可能产生不同大小写形式。

**修复方法**：使用不区分大小写的替换：

```swift
text = text.replacingOccurrences(
    of: wrong, with: correct,
    options: .caseInsensitive
)
```

### Bug 15-C：无法覆盖 Whisper 的所有错误变体

即使事后替换能工作，Whisper 对同一个词可能输出多种变体：

| 期望输出 | Whisper 可能的输出变体 |
|---------|---------------------|
| OpenAI | open ai, Open AI, openai, OPENAI, open-ai, OpenAl (L→l) |
| Claude | cloud, clod, Clade, 克劳德 |
| 李明 | 黎明, 离明, 里明 |

用户不可能手动穷举所有变体作为 alias。这是"事后替换"方案的本质局限——正确的解决路径是 Bug 15-A（让 Whisper 在解码时就知道这些词汇）。

**优先级**：最高（这是用户体验最核心的问题）
